1. EXECUTIVE SUMMARY
================================================================================

Arvos is an open-source iOS application that transforms your iPhone 12 Pro or
newer into a professional-grade multi-sensor research platform. It streams
real-time data from LiDAR, RGB camera, IMU (Inertial Measurement Unit), GPS,
ARKit pose tracking, and optional Apple Watch sensors over WiFi to any computer
for visualization, recording, and analysis.

Think of it as turning your iPhone into a mobile version of expensive research
equipment like Intel RealSense cameras ($300-500), Velodyne LiDAR sensors
($4,000-75,000), or Xsens IMU units ($2,000-10,000). All you need is an iPhone
you already own.

KEY CAPABILITIES:
- Real-time streaming at 30 FPS (camera), 10 FPS (LiDAR), 200 Hz (IMU)
- Sub-millisecond timestamp synchronization across all sensors
- Multiple streaming modes optimized for different research workflows
- Web-based visualization studio with 3D point cloud rendering
- MCAP format recording compatible with Foxglove, ROS, and research tools
- Zero-setup cloud relay option for immediate use
- Completely free and open source under GPL v3

WHO IS THIS FOR?
- Robotics researchers developing SLAM (Simultaneous Localization and Mapping)
- Computer vision students collecting training datasets
- AR/VR developers prototyping tracking algorithms
- Engineering teams testing sensor fusion algorithms
- Anyone needing affordable, portable, multi-sensor data collection

2. THE PROBLEM SPACE
================================================================================

RESEARCH HARDWARE IS EXPENSIVE AND INFLEXIBLE
----------------------------------------------
Traditional robotics and computer vision research requires specialized hardware:

- Intel RealSense D435i (RGB-D camera + IMU): $329
  Problem: Limited to 90° FOV, indoor-only depth sensing, bulky

- Velodyne Puck (LiDAR scanner): $7,999
  Problem: Heavy, power-hungry, requires mounting and calibration

- Xsens MTi-3 (IMU): $2,295
  Problem: Single-purpose device, needs integration with other sensors

- Survey-grade GPS: $3,000-15,000
  Problem: Expensive, cumbersome, overkill for most applications

SENSOR FUSION IS HARD
----------------------
Even if you buy individual sensors, integrating them is challenging:
- Different data rates (30 FPS camera vs 200 Hz IMU)
- Clock synchronization across devices
- Calibration between sensor coordinate systems
- Software drivers and data format conversion
- Physical mounting and power management

EXISTING SMARTPHONE SOLUTIONS ARE LIMITED
------------------------------------------
While apps exist to stream phone cameras, they typically:
- Only stream RGB video (no depth, IMU, or GPS)
- Lack nanosecond-precision timestamps needed for research
- Use proprietary formats incompatible with research tools
- Don't provide synchronized multi-sensor data
- Have no offline recording in standard formats

THE ARVOS SOLUTION
------------------
Arvos solves these problems by:
1. Leveraging iPhone's built-in professional sensors (LiDAR + ARKit + IMU + GPS)
2. Providing nanosecond-synchronized timestamps across all sensors
3. Using standard data formats (JSON, PLY point clouds, MCAP)
4. Offering both real-time streaming AND offline recording
5. Being completely free, open source, and modifiable
6. Working out-of-the-box with zero hardware setup

Result: A $1,000 iPhone becomes a $15,000+ mobile sensor platform.

3. WHAT IS ARVOS?
================================================================================

CORE CONCEPT
------------
Arvos is a native iOS application that runs on your iPhone and acts as a
WebSocket server. It continuously captures data from all available sensors,
timestamps it with nanosecond precision, and broadcasts it over your local
WiFi network.

Any device on the same network can connect and receive:
- RGB camera frames (1920x1080 JPEG)
- LiDAR depth point clouds (PLY format with confidence maps)
- IMU data (acceleration, angular velocity, gravity vectors)
- ARKit 6DOF pose (position and orientation in 3D space)
- GPS location (latitude, longitude, altitude)
- Apple Watch IMU (if paired watch is running companion app)

All sensor streams are synchronized to a common nanosecond timeline, making
them suitable for research-grade sensor fusion algorithms.

THE THREE COMPONENTS
---------------------

1. **Arvos iOS App** (this repository)
   - Native Swift application for iPhone 12 Pro or newer
   - Runs WebSocket server on port 8765
   - Captures and streams all sensor data
   - Optional companion app for Apple Watch
   - Records data to MCAP files directly on iPhone
   - Built with SwiftUI, ARKit, AVFoundation, CoreMotion

2. **Arvos Web Studio** (hosted on Vercel)
   - Browser-based visualization and control interface
   - Real-time 3D point cloud rendering with Three.js
   - Live camera feed display
   - IMU and GPS data charts
   - Recording controls and statistics
   - Works on any computer with a modern web browser
   - URL: https://arvos-studio-4u0khl359-jaskirat1616s-projects.vercel.app/

3. **Arvos Python SDK** (available on PyPI)
   - pip install arvos-sdk
   - Connect to Arvos streams programmatically
   - Process sensor data in real-time
   - Integration with Rerun visualization library
   - MCAP recording and playback
   - Perfect for custom algorithms and analysis

STREAMING MODES
---------------
Arvos provides pre-configured modes optimized for different workflows:

1. **RGBD Camera Mode**
   - Camera: 30 FPS | Depth: 30 FPS | IMU: OFF | Pose: OFF | GPS: OFF
   - Use Case: Acts like Intel RealSense or Kinect
   - Perfect for: 3D reconstruction, depth-based segmentation

2. **Visual-Inertial Mode**
   - Camera: 30 FPS | Depth: OFF | IMU: 200 Hz | Pose: OFF | GPS: OFF
   - Use Case: Visual-Inertial Odometry (VIO) development
   - Perfect for: SLAM algorithms, camera-IMU calibration

3. **LiDAR Scanner Mode**
   - Camera: OFF | Depth: 10 FPS | IMU: OFF | Pose: 10 Hz | GPS: OFF
   - Use Case: Pure 3D scanning with pose context
   - Perfect for: Point cloud mapping, environment modeling

4. **IMU Only Mode**
   - Camera: OFF | Depth: OFF | IMU: 200 Hz | Pose: OFF | GPS: OFF
   - Use Case: Pure inertial measurement
   - Perfect for: Motion tracking, activity recognition
   - Bonus: Enables Apple Watch IMU for dual-device tracking

5. **GPS Tracker Mode**
   - Camera: OFF | Depth: OFF | IMU: OFF | Pose: 10 Hz | GPS: ON
   - Use Case: Location tracking with orientation
   - Perfect for: Navigation, trajectory recording

6. **Full Sensor Mode**
   - Camera: 30 FPS | Depth: 10 FPS | IMU: 200 Hz | Pose: 60 Hz | GPS: ON
   - Use Case: Maximum data collection for research
   - Perfect for: Dataset creation, multi-sensor fusion
   - Includes: Apple Watch sensors automatically

7. **Low Power Mode**
   - Camera: 2 FPS | Depth: OFF | IMU: 50 Hz | Pose: OFF | GPS: OFF
   - Use Case: Extended battery life
   - Perfect for: Long-duration monitoring

8. **Custom Mode**
   - User selects: Any combination of sensors at custom rates
   - Use Case: Specific research requirements
   - Perfect for: Fine-tuned experiments

KEY ARCHITECTURAL DECISIONS
---------------------------

1. **Server-on-Device Architecture**
   - Why: Eliminates cloud dependencies and latency
   - Benefit: Works in labs, field, anywhere with WiFi
   - Trade-off: Requires local network connectivity

2. **WebSocket Protocol**
   - Why: Bidirectional, low-latency, widely supported
   - Benefit: Works in browsers, Python, ROS, any language
   - Alternative: gRPC, MQTT, QUIC adapters available

3. **Nanosecond Timestamps**
   - Why: Research-grade synchronization needed
   - Benefit: Accurate sensor fusion even at 200 Hz
   - Implementation: Uses mach_absolute_time() converted to ns

4. **Mixed Camera Systems**
   - ARKit camera for RGBD mode (integrated depth + RGB)
   - AVFoundation camera for camera-only modes (better control)
   - Automatic selection based on streaming mode

5. **MCAP Recording Format**
   - Why: Standard in robotics (Foxglove, ROS 2)
   - Benefit: Compatible with existing tools
   - Format: Self-describing, efficient binary storage

CORE MANAGERS
-------------

1. **SensorManager.swift**
   - Central coordinator for all sensors
   - Manages streaming modes and configurations
   - Routes sensor data to network and recording
   - Implements FPS throttling and rate limiting

2. **NetworkManager.swift**
   - Handles WebSocket server lifecycle
   - Manages client connections (supports multiple clients)
   - Protocol abstraction layer (WebSocket, gRPC, MQTT, etc.)
   - Broadcasts sensor data to all connected clients

3. **RecordingManager.swift**
   - MCAP file creation and management
   - Parallel H.264 video encoding for camera frames
   - Efficient binary serialization
   - On-device storage management

4. **WatchSensorManager.swift**
   - Manages Apple Watch connectivity
   - Receives IMU data from watch app
   - Synchronizes watch timestamps to iPhone timeline
   - Handles watch session lifecycle

5. **TimestampManager.swift**
   - Converts mach_absolute_time to nanosecond epochs
   - Ensures consistent timeline across all sensors
   - Handles timestamp drift correction

SENSOR SERVICES
---------------

1. **ARKitService.swift**
   - Configures ARSession for depth and pose tracking
   - Processes ARFrame data (depth maps, camera matrices)
   - Exports point clouds in PLY format
   - Manages ARFrame lifecycle to prevent memory leaks

2. **CameraService.swift**
   - AVCaptureSession configuration
   - 1920x1080 @ 30 FPS capture
   - JPEG compression with quality control
   - Intrinsic camera calibration export

3. **IMUService.swift**
   - CoreMotion DeviceMotion integration
   - 200 Hz accelerometer + gyroscope data
   - Gravity vector separation
   - Quaternion attitude representation

4. **GPSService.swift**
   - CoreLocation integration
   - Latitude, longitude, altitude
   - Accuracy and satellite status
   - Battery-efficient updates

5. SENSOR SYSTEMS DEEP DIVE
================================================================================

1. LiDAR DEPTH SENSOR
---------------------
**Hardware**: iPhone's rear-facing LiDAR scanner
**Technology**: Direct Time-of-Flight (dToF) measurement
**Specifications**:
  - Range: 0.5m to 5m
  - Resolution: 256x192 depth points
  - Frame Rate: Up to 10 FPS in Arvos
  - Accuracy: ±1% at typical ranges
  - Field of View: Matches camera FOV (~69° horizontal)

**How It Works in Arvos**:
  1. ARKit's ARSession captures ARFrame with depth data
  2. ARFrame contains sceneDepth (CVPixelBuffer in Float32 format)
  3. Each pixel represents distance in meters
  4. Confidence map indicates reliability (high/medium/low)
  5. Depth is converted to 3D point cloud using camera intrinsics
  6. Points are exported as PLY format with RGB color from camera
  7. Transmitted over WebSocket as binary message

**Data Format** (Point Cloud):
  ```
  PLY Binary Format
  Header:
    - Vertex count (int)
    - Property: x, y, z (float)
    - Property: r, g, b (uint8)
    - Property: confidence (float)

  Body:
    - Binary point data (typically 20,000-50,000 points per frame)
  ```

**Use Cases**:
  - 3D reconstruction of indoor environments
  - Obstacle detection for robotics
  - Depth-based segmentation
  - Spatial computing applications

2. RGB CAMERA
-------------
**Hardware**: iPhone's main wide-angle camera
**Specifications**:
  - Resolution: 1920x1080 (Full HD)
  - Frame Rate: 2-30 FPS (configurable)
  - Format: JPEG compressed (adjustable quality)
  - Color Space: sRGB
  - Lens: f/1.5, 26mm equivalent, 7-element

**Camera Systems**:
  Arvos uses two different camera systems:

  A. **ARKit Camera** (RGBD mode)
     - Integrated with depth capture
     - Synchronized depth + RGB frames
     - Automatic exposure and white balance
     - Camera intrinsics from ARCamera

  B. **AVFoundation Camera** (Camera-only modes)
     - Direct AVCaptureSession control
     - Better performance without ARKit overhead
     - Manual exposure control available
     - Lower latency

**Data Format** (Camera Frame):
  ```json
  {
    "type": "camera",
    "timestamp_ns": 1703001234567890123,
    "image_jpeg": "<base64 encoded JPEG>",
    "intrinsics": {
      "fx": 1344.5, // Focal length X
      "fy": 1344.5, // Focal length Y
      "cx": 960.0,  // Principal point X
      "cy": 540.0,  // Principal point Y
      "width": 1920,
      "height": 1080
    }
  }
  ```

**Optimizations**:
  - JPEG quality: 80% (balance between quality and bandwidth)
  - Frame skipping to maintain target FPS under load
  - Automatic rotation correction to landscape
  - Efficient memory management (CVPixelBuffer → CGImage → JPEG)

3. IMU (INERTIAL MEASUREMENT UNIT)
----------------------------------
**Hardware**: iPhone's built-in 6-axis IMU
**Components**:
  - 3-axis accelerometer (measures linear acceleration)
  - 3-axis gyroscope (measures angular velocity)
  - Sensor fusion by CoreMotion framework

**Specifications**:
  - Accelerometer Range: ±16g
  - Gyroscope Range: ±2000°/s
  - Update Rate: 50-200 Hz in Arvos
  - Noise: ~0.01 m/s² (accel), ~0.1°/s (gyro)

**Data Provided**:
  1. **Linear Acceleration** (user acceleration)
     - Acceleration in device reference frame
     - Gravity component removed
     - Units: m/s²
     - Used for: Motion tracking, gesture recognition

  2. **Angular Velocity** (rotation rate)
     - Rotation speed around X, Y, Z axes
     - Units: rad/s
     - Used for: Orientation tracking, stabilization

  3. **Gravity Vector**
     - Direction and magnitude of gravity
     - Separated from total acceleration
     - Used for: Leveling, orientation estimation

  4. **Attitude** (quaternion orientation)
     - Device orientation in 3D space
     - Quaternion representation (w, x, y, z)
     - Also available as Euler angles (pitch, roll, yaw)

**Data Format**:
  ```json
  {
    "type": "imu",
    "timestamp_ns": 1703001234567890123,
    "linear_acceleration": {
      "x": 0.15,  // m/s²
      "y": 0.02,
      "z": -0.08
    },
    "angular_velocity": {
      "x": 0.05,  // rad/s
      "y": -0.12,
      "z": 0.03
    },
    "gravity": {
      "x": 0.0,   // m/s²
      "y": -9.81,
      "z": 0.0
    },
    "attitude_quaternion": {
      "w": 0.98,
      "x": 0.01,
      "y": 0.15,
      "z": 0.02
    }
  }
  ```

**Coordinate System** (iPhone standard):
  - X-axis: Points to the right of the device
  - Y-axis: Points to the top of the device
  - Z-axis: Points out of the screen (right-handed)

4. ARKIT POSE TRACKING
----------------------
**What is ARKit?**
  ARKit is Apple's framework for Augmented Reality. It uses Visual-Inertial
  Odometry (VIO) to track the device's position and orientation in 3D space.

**How It Works**:
  1. Combines camera images with IMU data
  2. Detects and tracks visual features (corners, edges)
  3. Estimates 6DOF (6 Degrees of Freedom) pose:
     - Position: X, Y, Z in meters
     - Orientation: Roll, pitch, yaw (or quaternion)
  4. Continuously refines estimates as device moves

**Specifications**:
  - Update Rate: 30-60 Hz
  - Positional Accuracy: ±1-2 cm in good lighting
  - Rotational Accuracy: ±1°
  - Tracking Range: Indoor environments up to room scale
  - Recovery: Automatically relocates after tracking loss

**Data Format**:
  ```json
  {
    "type": "pose",
    "timestamp_ns": 1703001234567890123,
    "position": {
      "x": 1.23,  // meters
      "y": 0.45,
      "z": -2.10
    },
    "orientation_quaternion": {
      "w": 0.97,
      "x": 0.02,
      "y": 0.24,
      "z": 0.01
    },
    "transform_matrix": [
      [0.98, -0.15, 0.12, 1.23],
      [0.16,  0.98, -0.08, 0.45],
      [-0.11, 0.09, 0.99, -2.10],
      [0.00,  0.00, 0.00, 1.00]
    ]
  }
  ```

**Use Cases**:
  - Ground truth for SLAM algorithm evaluation
  - Robot localization
  - AR content placement
  - Trajectory recording

**Limitations**:
  - Requires good lighting (struggles in darkness)
  - Needs visual texture (blank walls cause drift)
  - Scale may drift over long distances
  - Coordinate system resets when app restarts

5. GPS LOCATION
---------------
**Hardware**: iPhone's GPS, GLONASS, Galileo, QZSS receiver
**Specifications**:
  - Accuracy: ±5-10m (typical), ±3m with good satellite lock
  - Update Rate: 1 Hz in Arvos
  - Cold Start: 30-60 seconds
  - Warm Start: 5-10 seconds

**Data Provided**:
  - Latitude (decimal degrees)
  - Longitude (decimal degrees)
  - Altitude (meters above sea level)
  - Horizontal Accuracy (uncertainty radius)
  - Vertical Accuracy
  - Speed (m/s)
  - Course (heading in degrees)

**Data Format**:
  ```json
  {
    "type": "gps",
    "timestamp_ns": 1703001234567890123,
    "latitude": 37.7749,       // degrees
    "longitude": -122.4194,    // degrees
    "altitude": 52.3,          // meters
    "horizontal_accuracy": 5.2, // meters
    "vertical_accuracy": 8.1,   // meters
    "speed": 1.2,              // m/s
    "course": 245.5            // degrees
  }
  ```

**Optimization**:
  - Uses kCLLocationAccuracyBest for maximum precision
  - Battery-efficient update filtering
  - Only streams when GPS lock acquired

**Use Cases**:
  - Outdoor navigation and mapping
  - Geo-tagging point clouds and images
  - Trajectory recording for vehicles
  - Location-based research

6. NETWORK ARCHITECTURE & PROTOCOLS
================================================================================

WEBSOCKET SERVER (DEFAULT)
--------------------------
**Why WebSocket?**
  - Bidirectional real-time communication
  - Low latency (~5-20ms on local WiFi)
  - Wide language support (JavaScript, Python, C++, ROS)
  - Works in web browsers without plugins

**Implementation**:
  - Built on Apple's Network framework (NWConnection)
  - Server runs on iPhone at port 8765
  - Supports multiple simultaneous clients
  - Automatic client connection/disconnection handling

**Connection Flow**:
  1. iPhone app starts → WebSocket server launches on port 8765
  2. App displays connection URLs:
     - WiFi: ws://192.168.1.100:8765
     - Hotspot: ws://172.20.10.1:8765
  3. Client (Web Studio or Python SDK) connects to URL
  4. Handshake completes, connection established
  5. Server begins broadcasting sensor data to all clients
  6. Client can send control commands (start recording, change mode, etc.)

**Message Types**:

  A. **JSON Messages** (sensor data, control)
     - IMU data
     - GPS data
     - Pose data
     - Mode configuration
     - Statistics

  B. **Binary Messages** (large data)
     - Camera frames (JPEG)
     - Depth point clouds (PLY)
     - Video chunks (H.264)

**Protocol Structure**:
  ```
  Message Format:
  [TYPE: 1 byte][PAYLOAD: variable]

  Type 0x01: JSON message
  Type 0x02: Binary camera frame
  Type 0x03: Binary point cloud
  Type 0x04: Binary video chunk
  ```

ALTERNATIVE PROTOCOLS (PLANNED/AVAILABLE)
-----------------------------------------
Arvos includes a protocol abstraction layer supporting:

1. **gRPC**
   - High-performance RPC framework
   - Efficient binary protocol (Protocol Buffers)
   - Ideal for: Research platforms, ROS 2 integration
   - Port: 50051

2. **MQTT**
   - Publish-subscribe messaging
   - Multiple subscribers to same topic
   - Ideal for: IoT applications, multi-client scenarios
   - Port: 1883

3. **QUIC / HTTP3**
   - Modern low-latency protocol
   - Built-in encryption
   - Ideal for: High-frequency data, mobile networks
   - Port: 4433

4. **MCAP Stream**
   - Direct MCAP file streaming
   - Native Foxglove format
   - Ideal for: ROS 2 integration, robotics research
   - Port: 17500

5. **HTTP/REST**
   - Simple request-response
   - Universal compatibility
   - Ideal for: Simple integrations, debugging
   - Port: 8080

6. **Bluetooth LE**
   - Cable-free, low bandwidth
   - Ideal for: Wearable research, close-range
   - No port (BLE)

CONNECTION MODES
----------------

1. **Direct WiFi** (Default)
   - iPhone and computer on same WiFi network
   - Lowest latency (~5-10ms)
   - Best for: Lab environments, home networks
   - Setup: Connect both devices to same WiFi

2. **iPhone Hotspot**
   - iPhone creates WiFi hotspot
   - Computer connects to iPhone's network
   - Best for: Field work, no WiFi available
   - Setup: Enable Personal Hotspot on iPhone
   - URL: ws://172.20.10.1:8765

3. **Cloud Relay** (Optional)
   - iPhone → Cloud server ← Computer
   - Works over internet (different networks)
   - Higher latency (~50-200ms)
   - Best for: Remote collaboration, demos
   - Setup: Tap "Use Cloud Relay" in app
   - Server: arvos-web.onrender.com (free tier)

BANDWIDTH CHARACTERISTICS
--------------------------
Typical data rates per sensor (Full Sensor mode):

- Camera (30 FPS, JPEG):  ~5-8 Mbps
- Depth (10 FPS, PLY):    ~2-4 Mbps
- IMU (200 Hz, JSON):     ~0.05 Mbps
- Pose (60 Hz, JSON):     ~0.02 Mbps
- GPS (1 Hz, JSON):       ~0.001 Mbps
- Watch (50 Hz, JSON):    ~0.03 Mbps

Total: ~7-12 Mbps (well within WiFi capacity of 50-100+ Mbps)

Quality of Service:
- Local WiFi: Excellent (10-20ms latency, 0% packet loss)
- Hotspot: Very Good (15-30ms latency, <0.1% packet loss)
- Cloud Relay: Good (50-200ms latency, <1% packet loss)

7. DATA PIPELINE & SYNCHRONIZATION
================================================================================

TIMESTAMP SYNCHRONIZATION
-------------------------
The most critical challenge in multi-sensor systems is synchronization. When
you have a camera at 30 FPS, IMU at 200 Hz, and LiDAR at 10 FPS, you need a
common timeline to fuse data correctly.

**Arvos Solution: Nanosecond Epoch Timestamps**

All sensors use mach_absolute_time(), a monotonic clock that never goes
backward (unlike system time which can change with NTP sync or user adjustment).

Implementation (TimestampManager.swift):
```swift
func now() -> UInt64 {
    let machTime = mach_absolute_time()
    let nanos = machTime * timebase.numer / timebase.denom
    return nanos
}
```

This gives nanosecond precision timestamps like:
  1703001234567890123

Benefits:
- Sub-millisecond accuracy (nanosecond resolution)
- Monotonic (never jumps backward)
- Consistent across all sensors
- Compatible with ROS, Foxglove, research tools

**Synchronization Guarantees**:
- IMU samples: <0.5ms jitter between samples
- Camera frames: Timestamped at exact capture moment
- Depth frames: Synchronized with camera when using ARKit
- GPS: Timestamped at location update callback
- Watch: Timestamps converted from watch timeline to iPhone timeline

DATA FLOW PIPELINE
-------------------

Step-by-step journey of sensor data:

1. **CAPTURE**
   ```
   Sensor Hardware → iOS Framework Callback

   Examples:
   - ARKit: ARSessionDelegate didUpdate(frame)
   - Camera: AVCaptureVideoDataOutputSampleBufferDelegate
   - IMU: CMMotionManager deviceMotionUpdateHandler
   - GPS: CLLocationManagerDelegate didUpdateLocations
   ```

2. **TIMESTAMP**
   ```
   Callback → TimestampManager.now() → Nanosecond timestamp

   Timestamp attached to raw sensor data immediately
   ```

3. **PROCESS**
   ```
   Raw Data → Service (ARKitService, CameraService, etc.)

   Processing:
   - ARKit: Extract depth map, convert to point cloud
   - Camera: Convert CVPixelBuffer → JPEG
   - IMU: Extract acceleration, angular velocity, quaternion
   - GPS: Extract lat/lon/altitude
   ```

4. **ROUTE**
   ```
   Service → SensorManager (via delegate pattern)

   SensorManager checks:
   - Is this sensor enabled in current mode?
   - Is data rate within FPS limit?
   - Are we currently streaming?
   ```

5. **DISTRIBUTE**
   ```
   SensorManager → NetworkManager + RecordingManager (parallel)

   Branching:
   - NetworkManager: Send to WebSocket clients
   - RecordingManager: Write to MCAP file (if recording)
   ```

6. **SERIALIZE**
   ```
   Data Object → Wire Format

   JSON sensors: Swift Codable → JSON string
   Binary sensors:
     - Camera: JPEG bytes
     - Depth: PLY format bytes
   ```

7. **TRANSMIT**
   ```
   Wire Format → WebSocket.send() → Network → Clients

   Delivery:
   - JSON: WebSocket text message
   - Binary: WebSocket binary message
   - Broadcast to all connected clients simultaneously
   ```

8. **RECEIVE & VISUALIZE**
   ```
   Client → Parse → Render

   Web Studio:
   - Camera: Display JPEG in <img> element
   - Depth: Parse PLY → Three.js point cloud → WebGL
   - IMU: Update real-time charts
   - Pose: Update 3D camera frustum visualization

   Python SDK:
   - Parse JSON/binary
   - Route to Rerun visualization
   - Or custom processing callback
   ```

FRAME RATE CONTROL
-------------------
To ensure consistent FPS and prevent overwhelming the network:

1. **FPS Throttling**
   ```swift
   // Only send if enough time elapsed since last frame
   let now = Date()
   let elapsed = now.timeIntervalSince(lastCameraFrameTime)
   if elapsed < 1.0 / Double(targetFPS) {
       return // Skip this frame
   }
   ```

2. **Burst Protection**
   - ARKit can deliver frames faster than target rate
   - Arvos drops intermediate frames to maintain consistent FPS
   - Ensures predictable bandwidth usage

3. **Adaptive Quality**
   - JPEG quality can be adjusted (50-100%)
   - Point cloud decimation for lower bandwidth
   - Automatic downsampling under network pressure

MEMORY MANAGEMENT
-----------------
Streaming from multiple sensors simultaneously can strain iPhone memory:

**ARFrame Retention Prevention**:
- ARFrames are large (depth map + camera image + metadata)
- Must be released immediately after processing
- Arvos uses explicit frame capture and immediate release
- Monitoring: Debug logs warn if ARFrame count exceeds threshold

**Buffer Pools**:
- Reuse CVPixelBuffers where possible
- Pre-allocate JPEG compression buffers
- Efficient point cloud memory layout

**Pressure Handling**:
- Monitor system memory pressure notifications
- Reduce quality or FPS under high pressure
- Graceful degradation instead of crashes

MEMORY MANAGEMENT STRATEGIES
-----------------------------

**ARFrame Lifecycle**
ARFrames are retained by ARKit and can cause memory pressure if not released:

```swift
// ❌ BAD - Retains ARFrame
self.latestDepthSample = frame.sceneDepth

// ✅ GOOD - Extract data and release frame
func processDepthFrame(_ frame: ARFrame) {
    guard let depthMap = frame.sceneDepth else { return }

    // Extract point cloud data
    let points = convertDepthToPointCloud(depthMap)

    // Store processed data, not ARFrame
    self.latestDepthFrame = DepthFrame(
        points: points,
        timestamp: TimestampManager.now()
    )

    // ARFrame released when function exits
}
```

Monitoring:
```swift
let retainCount = CFGetRetainCount(frame)
if retainCount > 10 {
    DebugLogger.log("⚠️ ARFrame retention issue detected")
}
```

**Buffer Reuse**
For high-frequency data like IMU:

```swift
// Reuse pre-allocated buffers instead of creating new objects
var imuDataPool: [IMUData] = (0..<100).map { _ in IMUData() }
var poolIndex = 0

func handleIMUUpdate(_ motion: CMDeviceMotion) {
    let data = imuDataPool[poolIndex]
    data.update(from: motion)
    sendToNetwork(data)
    poolIndex = (poolIndex + 1) % imuDataPool.count
}
```

THREADING MODEL
---------------

Arvos uses multiple threads to prevent blocking:

1. **Main Thread**
   - UI updates only
   - SwiftUI view rendering
   - User interaction handling

2. **ARKit Thread** (managed by ARSession)
   - ARFrame delivery
   - Depth processing
   - Pose estimation

3. **Camera Thread** (AVFoundation queue)
   - Camera frame capture
   - JPEG compression

4. **IMU Thread** (CoreMotion queue)
   - High-frequency IMU updates (200 Hz)
   - Attitude calculations

5. **Network Thread** (NWConnection queue)
   - WebSocket message sending
   - Client connection handling

6. **Recording Thread** (Background queue)
   - MCAP file writing
   - Video encoding

Thread Safety:
- Use @Published properties with @MainActor for UI
- Serial queues for critical sections
- Concurrent queues for parallel processing
- Locks only where necessary (prefer actor model)

```swift
func startStreaming() {
```

PERFORMANCE OPTIMIZATIONS
--------------------------

1. **JPEG Compression**
   - Parallel compression using DispatchQueue.concurrentPerform
   - Quality: 80% (balance file size vs visual quality)
   - Hardware acceleration when available

2. **Point Cloud Decimation**
   - Skip points below confidence threshold
   - Spatial downsampling for distant points
   - Reduces bandwidth without losing detail

3. **JSON Minimization**
   - Short key names ("ts_ns" vs "timestamp_nanoseconds")
   - Remove whitespace (no pretty-printing)
   - Float precision limited to 6 decimals

4. **Binary Protocol**
   - Camera and depth sent as binary (not base64)
   - 30% smaller than base64 encoding
   - Faster parsing on client side

5. **Lazy Initialization**
   - Services only start when needed
   - GPS not initialized until GPS mode selected
   - ARKit session delayed until first use

This section shares the story behind Arvos - why it was built, challenges

MOTIVATION
----------
The project started from a simple question: "Why do robotics researchers spend
$10,000+ on sensors when iPhones have professional-grade LiDAR, IMU, and
cameras?"

The iPhone 12 Pro (released 2020) was revolutionary - it included a direct
time-of-flight LiDAR sensor typically found in $300+ depth cameras. Combined
with ARKit's world tracking and high-frequency IMU, it matched or exceeded
many research platforms.

But there was no good way to stream this data off the iPhone for external
processing. Existing apps either:
- Only streamed camera (no depth or IMU)
- Used proprietary formats
- Lacked timestamp synchronization
- Required cloud accounts or subscriptions

Arvos was created to fill this gap: a professional, open-source, multi-sensor
streaming platform using standard formats and protocols.

DESIGN PHILOSOPHY
-----------------

1. **Open Source First**
   - GPL v3 license (copyleft)
   - No proprietary SDKs or cloud lock-in
   - Forkable and modifiable
   - Community-driven development

2. **Standards-Based**

   - JSON (universal parsing)

   - MCAP recording (robotics standard)

3. **Zero Setup**

   - Pre-configured streaming modes

4. **Research Grade**

5. **Developer Friendly**

TECHNICAL CHALLENGES SOLVED
---------------------------

1. **ARFrame Memory Leaks**
   Problem: ARKit retains ARFrames aggressively. Holding references caused
            memory warnings and app crashes after 30 seconds.
   Solution: Extract data immediately (point cloud, pose) and never store
             ARFrame itself. Monitor retain count in debug builds.

2. **Timestamp Synchronization**
   Problem: Camera uses CMTime, IMU uses Date, ARKit uses CFAbsoluteTime.
            How to synchronize?
   Solution: Use mach_absolute_time() for everything. Convert all sensor
             timestamps to nanosecond epoch. Sub-millisecond sync achieved.

3. **WebSocket Server on iOS**
   Problem: iOS doesn't have built-in WebSocket server APIs.
   Solution: Build custom server using Network framework (NWListener).
             Handle WebSocket handshake, framing, and multiple clients.

4. **Camera + Depth Synchronization**
   Problem: AVFoundation camera and ARKit depth have different capture times.
   Solution: Use ARKit's camera when both needed (RGBD mode). ARFrame
             contains synchronized camera + depth. Use AVFoundation only
             for camera-only modes.

5. **High-Frequency IMU Without Blocking**
   Problem: 200 Hz IMU generates 200 callbacks/second. Processing on main
            thread blocks UI.
   Solution: Dedicated serial queue for IMU. Batch processing every 5 samples.
             Async network send without waiting for completion.

6. **Point Cloud Compression**
   Problem: Raw depth is 256x192 Float32 = 200 KB/frame. At 10 FPS = 2 MB/s.
   Solution: Export as PLY (efficient binary format). Filter low-confidence
             points. Compress with zlib (optional). ~30 KB/frame typical.

7. **WiFi Discovery**
   Problem: Users don't know their iPhone's IP address.
   Solution: Automatic IP detection (enumerate network interfaces). Display
             QR code for easy scanning. Support both WiFi and Hotspot IPs.

8. **Battery Life**
   Problem: LiDAR + camera + ARKit drains battery in 90 minutes.
   Solution: Streaming modes (disable unneeded sensors). Low Power mode.
             Efficient threading (no busy waiting). User education.

9. **Apple Watch Integration**
   Problem: Watch Connectivity is complex, messages can be lost.
   Solution: Reliable message queue with retry logic. Timestamp synchronization
             using round-trip time measurement. Graceful degradation if
             watch disconnects.

10. **Cross-Platform Visualization**
    Problem: Native 3D rendering on web is hard.
    Solution: Built separate Web Studio using Three.js. Progressive enhancement
              (works without WebGL, better with). Falls back to 2D if GPU
              unavailable.

LESSONS LEARNED
---------------

1. **Memory Management is Critical**
   iOS will terminate apps that use too much memory. For sensor streaming,
   you MUST release buffers immediately. Don't hold references to ARFrames,
   CVPixelBuffers, or large objects.

2. **Threading is Your Friend**
   Don't block the main thread. High-frequency sensors (IMU, camera) need
   dedicated threads. Use serial queues for order, concurrent for parallelism.

3. **Error Handling is Not Optional**
   Sensors fail (camera permission denied, GPS unavailable). Handle gracefully.
   Continue with working sensors. Inform user clearly.

4. **Performance Testing on Real Hardware**
   Simulators don't have LiDAR, and performance is unrealistic. Always test
   on actual iPhone. Test in hot/cold environments. Test with low battery.

5. **Documentation Beats Clever Code**
   Six months later, you won't remember why you did something. Write comments.
   Explain the "why", not just the "what". Future you will thank you.

6. **Open Source Attracts Collaboration**
   Publishing on GitHub led to bug reports, feature requests, and contributions.
   Users from robotics labs, AR companies, and universities provided feedback
   that shaped the project.

7. **Standards Matter**
   Using WebSocket, JSON, PLY, and MCAP meant instant compatibility with
   existing tools. Don't invent proprietary formats unless absolutely necessary.

**Project 2: 3D Reconstruction**

```python
# Build point cloud map
from arvos_sdk import ArvosClient
import open3d as o3d

client = ArvosClient("192.168.1.100:8765")
client.connect()

global_map = o3d.geometry.PointCloud()

for data in client.stream():
    if data["type"] == "depth":
        # Convert to Open3D point cloud
        pcd = o3d.geometry.PointCloud()
        pcd.points = o3d.utility.Vector3dVector(data["points"])
        pcd.colors = o3d.utility.Vector3dVector(data["colors"])

        # Get current pose
        transform = data.get("transform_matrix")
        if transform:
            pcd.transform(transform)

        # Merge into global map
        global_map += pcd

        # Downsample to prevent memory overflow
        global_map = global_map.voxel_down_sample(0.02)

# Save final map
o3d.io.write_point_cloud("map.ply", global_map)
```

TROUBLESHOOTING
---------------

**Problem: "Cannot connect to iPhone"**
Solutions:
- Ensure iPhone and computer on same WiFi network
- Check firewall allows port 8765
- Try iPhone's hotspot instead
- Use "Cloud Relay" option as fallback

**Problem: "Camera permission denied"**
Solutions:
- Go to iPhone Settings → Privacy → Camera → Arvos → Enable
- Restart app after granting permission

**Problem: "Low frame rate / stuttering"**
Solutions:
- Close other apps on iPhone
- Reduce sensor rates (use Custom mode)
- Check WiFi signal strength
- Try lower camera resolution

**Problem: "App crashes after 30 seconds"**
Solutions:
- Update to latest version (ARFrame memory leak fixed)
- Restart iPhone
- Clear storage (Settings → General → iPhone Storage)

**Problem: "No depth data shown"**
Solutions:
- Requires iPhone 12 Pro or newer (LiDAR)
- Enable depth in streaming mode
- Check lighting (LiDAR struggles in direct sunlight)

**Problem: "Web Studio not loading"**
Solutions:
- Try different browser (Chrome recommended)
- Disable browser extensions
- Clear cache and reload
- Check browser console for errors

CALIBRATION DATA
----------------

Arvos provides camera intrinsics with every frame:

```python
{
  "intrinsics": {
    "fx": 1344.5,  # Focal length X (pixels)
    "fy": 1344.5,  # Focal length Y (pixels)
    "cx": 960.0,   # Principal point X (pixels)
    "cy": 540.0,   # Principal point Y (pixels)
    "width": 1920,
    "height": 1080
  }
}
```

This matches the standard pinhole camera model:
```
x_image = fx * X_camera / Z_camera + cx
y_image = fy * Y_camera / Z_camera + cy
```

Distortion coefficients not provided (iPhone has minimal lens distortion
after iOS processing).

COORDINATE SYSTEMS
------------------

**IMU Coordinate System** (iPhone Standard):
- X: Right
- Y: Up
- Z: Out of screen
- Right-handed

**ARKit Coordinate System** (World Tracking):
- X: Right
- Y: Up
- Z: Backward (into screen)
- Right-handed
- Origin: Where device was when ARSession started

**Camera Coordinate System**:
- Same as ARKit when using ARKit camera
- Image origin: Top-left corner
- +X: Right, +Y: Down

**Conversions**:
```python
# ARKit to ROS (REP-103)
def arkit_to_ros(position):
    return {
        "x": position["z"],   # Forward
        "y": -position["x"],  # Left
        "z": position["y"]    # Up
    }
```

```swift
// MagnetometerService.swift
import CoreLocation

class MagnetometerService {

        locationManager.startUpdatingHeading()

            x: heading.x,
            y: heading.y,
            z: heading.z

    }
}
```

```swift
let magnetometerService = MagnetometerService()

func startStreaming() {
```

```python
recorder.write(data)
recorder.stop()

# Disconnect
client.disconnect()
```

PERFORMANCE OPTIMIZATION
-------------------------
1. Use WiFi 5 or 6 (not 4)
2. Close background apps
3. Reduce camera FPS (30→15)
4. Disable unneeded sensors (Custom mode)
5. Lower JPEG quality if bandwidth limited
6. Use hotspot if WiFi congested
7. Keep iPhone cool (avoid direct sun)
8. Charge while streaming for long sessions
